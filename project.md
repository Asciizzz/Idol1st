# Idol1st — Project Documentation

> Generated by reading the codebase directly (routes, controllers, middleware, models, migrations, views, checklists, and `Refactor.md`). Reflects the state of the code as uploaded, not just what the docs claim.

## 1. What This Project Is

Idol1st is a **multi-tenant SaaS platform** with two products bolted together:

1. **A visual, node-graph website builder ("VirtualSiteBuilder / VSB")** — an editor where a user (an "idol"/creator or their team) drags nodes on a graph to design a website, then compiles and publishes it. This is the original product (see `Checklists/Draft/CHECKLIST - Phase 1.md`).
2. **A hosted "idol fan site" SaaS layer** built on top of tenants — each tenant gets a subdomain (e.g. `sakura.idol1st.test`) with a fan-facing site offering blog/news, events, paid membership tiers, and a merch store. This is the newer, larger layer (Steps 8–17 in `Checklists/`).

Stack: **Laravel 12** (PHP 8.2+), **MySQL**, **Sanctum** for API tokens, a **custom vanilla-JS** node-graph editor (no Vue/React — `resources/js/VirtualSiteBuilder`, `resources/js/Alib/{Awgl2,Awgpu,VsGraph}` are hand-rolled WebGL/graph libraries), Vite for bundling, Vitest for JS tests, and Stripe/PayPal SDKs for payments.

## 2. Tenancy Model

- **Tenant** = one idol's site (`tenants` table, UUID PK, `slug` used for subdomain routing, `status` of `ACTIVE`/`SUSPENDED`, JSON `config`).
- Tenant resolution happens per-request in `ResolveTenant` middleware: it first tries to resolve tenant from the Sanctum bearer token's owner (`tokenable->tenant_id`), then falls back to the subdomain (`{slug}.idol1st.test`). Reserved subdomains (`idol1st`, `www`, `localhost`, `127`) are rejected as "no tenant context."
- **Users vs Fans are two separate identity systems:**
  - `users` table — platform/editor accounts. A user can also be a **tenant admin** via `is_tenant_admin` + `tenant_id` columns (this was merged from a formerly-separate `tenant_admins` table — see `Refactor.md`; the old `TenantAdmin` model/migration-drop is dead code still sitting in `app/Models/TenantAdmin.php`).
  - `fans` table — separate auth guard (`auth:fan`) for end customers on a tenant's public site (login, membership, merch, blog comments, RSVPs).
  - `service_admins` table — a third identity, for **platform-level** staff (Idol1st's own operators), gated by `EnsureServiceAdmin` and supports MFA (Google2FA).
- Projects, blog posts, events, merch, membership tiers, etc. all carry `tenant_id` and are scoped to the resolved tenant. `Project` additionally enforces this via a **global Eloquent scope** (`booted()` in `app/Models/Project.php`) that filters every query by the current request's tenant.

## 4. Middleware

| Middleware | Purpose |
|---|---|
| `ResolveTenant` | Resolves `Tenant` from bearer token owner or subdomain; rejects reserved hosts; blocks suspended tenants (403); attaches `$request->tenant()` and binds `Tenant::class` in the container. |
| `EnsureFan` | Requires an authenticated `fan` guard user; redirects to tenant login (web) or 401 (JSON); 403s if the fan's `tenant_id` doesn't match the resolved tenant. |
| `EnsureTenantAdmin` | Requires `$user->is_tenant_admin` and matching `tenant_id` against the resolved tenant. |
| `EnsureServiceAdmin` | Requires the authenticated user to be a `ServiceAdmin` instance and `is_active`. |
| `EnsureAdmin` | Requires `$user->role` in `['ADMIN', 'SUPER_ADMIN']` — appears to be legacy/unused by current routes (superseded by `EnsureServiceAdmin`/`is_tenant_admin`); worth confirming it's not dead code. |
| `RequireAuth` | Session-based gate for Blade routes (`/editor`, `/admin`). Supports an optional `:admin` parameter that also accepts a `service_admin_sanctum_token` stored in session, bridging session-auth pages to a Sanctum-token-holding service admin. |

## 5. Controllers (31)

- **Root:** `AuthEditorController` (editor login/session/API auth), `AdminController` (legacy admin stats/users/projects endpoints — overlaps with `Admin\*`), `AssetController`, `CompilerController` (compiles VSB graph → HTML/CSS/JS), `ProjectController`, `PublishController`, `SnapshotController`, `FanSiteController` (renders the public Blade fan site), `PaymentWebhookController`.
- **`Admin/*`** (platform staff): `PlatformAdminAuthController`, `TenantController` (CRUD + suspend/reactivate/impersonate/assign-plan), `PlanController`, `FeatureFlagController` (with per-tenant rollout overrides), `AuditLogController`.
- **`Manage/*`** (tenant admin, i.e. the idol's own back office): `TenantAdminAuthController`, `IdolProfileController`, `IdolGroupController`, `SocialLinkController`, `BlogController`, `EventController`, `MembershipTierController`, `MerchController` (products + order fulfillment/shipping).
- **`Fan/*`** (public/customer-facing API): `FanAuthController`, `FanProfileController`, `AddressController`, `BlogController` (read + comment + like), `EventController` (read + RSVP), `FanSubscriptionController` (membership subscribe/upgrade/cancel), `MerchController` (cart/checkout/orders), `NotificationController`.

## 6. Views (Blade)

Two rendering surfaces:
- **App shell views** (`admin.blade.php`, `tenant-panel.blade.php`, `login.blade.php`) — thin Blade wrappers that boot the JS apps (`AdminPanel`, `VirtualSiteBuilder`) via Vite.
- **Fan site** (`resources/views/fan/*`) — server-rendered per-tenant pages: `layout`, `home`, `news`, `events`, `membership`, `merch`, `cart`, `checkout`, `order-success`, `login`. These are classic Blade views (not SPA), rendered by `FanSiteController`, while the same data is also exposed via the `Fan\*` JSON API — so there are effectively two front-ends for the fan experience.

## 7. Database (40 migrations)

Core domains:
- **Identity/tenancy:** `users` (+ `role`, `tenant_id`, `is_tenant_admin` added later), `tenants` (+ `slug` added later), `tenant_subscriptions`, `plans`, `service_admins`, `fans`.
- **Editor/builder:** `projects` (+ `tenant_id` backfilled later), `project_snapshots` (versioned graph/compiled output), `assets`, `published_sites`.
- **Platform ops:** `feature_flags` (+ tenant pivot), `audit_logs`.
- **Idol content:** `idol_profiles`, `idol_groups` (+ pivot to profiles), `social_links`.
- **Blog:** `blog_categories`, `blog_posts`, `blog_comments`, (likes via a model without its own migration file listed separately — `BlogPostLike`).
- **Membership:** `membership_tiers`, `tier_perks` (model exists; migration presumably part of the tier migration), `fan_subscriptions`.
- **Commerce:** `merch_products`, `merch_categories`, `merch_variants`, `merch_carts`, `merch_cart_items`, `merch_orders`, `merch_order_items`, `payments`, `shipments`, `addresses`.
- **Events:** `idol_events`, `event_rsvps`.
- **Notifications:** `fan_notifications`, `notification_preferences`.
- **Framework:** `sessions`, `cache`, `personal_access_tokens` (Sanctum; `tokenable_id` widened to string to support UUID tenants/fans).

**Migration history worth knowing:** the schema was reworked mid-project (`2026_06_30_*` migrations) to merge `tenant_admins` into `users`, add `slug` to `tenants`, and backfill `tenant_id` onto `projects` — i.e. this was originally a single-tenant editor that was retrofitted into multi-tenant SaaS. The `TenantAdmin` **model** still exists in `app/Models/TenantAdmin.php` even though its table was dropped — this is dead code and will error if instantiated/queried.

Almost every domain table uses **UUID primary keys** (`HasUuids`) except the original `users`/`projects` tables, which still use auto-increment bigints (`HasFactory` only) — a schema inconsistency to be aware of when writing joins or API responses.

## 8. What Currently Works

Based on what has real controller + route + migration + (for most) checklist coverage:

- **Multi-tenant resolution** by subdomain or token, with suspension handling.
- **Three separate auth flows**: editor/user (Sanctum), fan (Sanctum, `fan` guard), platform service admin (Sanctum + MFA via Google2FA).
- **Visual site builder**: create/list/update/delete projects, save versioned snapshots, compile a project, publish it, and view the published output.
- **Platform admin console**: manage tenants (create, suspend/reactivate, assign plan, impersonate), plans, feature flags (with per-tenant rollout), and audit logs.
- **Tenant admin back office**: manage idol profile + social links + groups, write/publish blog posts, moderate comments, create events, manage membership tiers, manage merch catalog and fulfill/ship orders.
- **Fan-facing features**: register/login, read blog + comment + like, view/RSVP events, subscribe/upgrade/cancel membership, browse merch, cart, checkout, view/cancel orders, manage profile + addresses, notification preferences.
- **Payments**: Stripe and PayPal SDKs are wired in, with webhook handlers per provider (`StripeWebhookHandler`, `PaypalWebhookHandler`, `QrPaymentWebhookHandler`) dispatched through `PaymentWebhookDispatcher` at the public `/api/webhooks/payment/{provider}` endpoint. The route also lists several QR-based providers (`promptpay`, `duitnow`, `qris`, `paynow`, `instapay`, `wechatpay`, `alipay`) though only Stripe/PayPal packages are actually in `composer.json`.

## 9. What's Lacking / Incomplete

- **Custom domains for tenants** — `PublishedSite.domain` column exists but there's no domain-verification or DNS/SSL flow; only subdomain routing is implemented.
- **Tenant onboarding & billing UI** — tenant/plan management exists on the *platform admin* side, but there's no self-serve signup, plan upgrade, or billing portal for tenants themselves. `Refactor.md` explicitly lists "Billing integration" and "Tenant settings management" as future work.
- **Background jobs / queue workers** — `QUEUE_CONNECTION=database` is set but there are no `app/Jobs`, `app/Events`, or `app/Listeners` in the codebase. Notifications, order emails, and webhook side-effects all appear to run synchronously inline in controllers/services rather than being queued.
- **Outbound email** — `MAIL_MAILER=log`; no transactional email (order confirmation, password reset, etc.) is actually configured to send anywhere but the log file, and no `Mailable` classes were found.
- **Dead/duplicate code from the tenancy retrofit**: `App\Models\TenantAdmin` still references a dropped table; `AdminController` (root-level) appears to duplicate functionality now under `Admin\*` controllers — should be reconciled or removed.
- **Ambiguous `/api/auth/logout` route** — registered under two different route groups/guards; needs to be split or clarified (see `ROUTES.md` notes).
- **Diagnostic/test routes left in place** (`api/admin/test-plans`, `api/admin/test-auth`, `api/tenant-test`) — need to be removed or hidden behind an environment check before production.
- **Two parallel fan front-ends** — the server-rendered Blade fan site (`resources/views/fan/*`) and the JSON `Fan\*` API overlap in responsibility; unclear which is the long-term surface (likely the API, with Blade as an early/interim implementation), which could cause drift.
- **No automated backend test suite observed** — only `vitest` (JS) is configured; no PHPUnit/Pest tests were found for controllers, middleware, or models, so regressions in tenant isolation or auth boundaries wouldn't be caught automatically.
- **Schema inconsistency** — mixed primary key strategies (UUID vs bigint) across tables, which is fine functionally but worth standardizing for long-term maintainability.
- **`EnsureAdmin` middleware appears unused** by current routes (superseded by `EnsureServiceAdmin`/`is_tenant_admin` checks) — likely safe to remove, but should be confirmed before deleting.

## 10. Cross-Check Against `Idol_SaaS_Full_System.puml`

The repo also contains a full UML class diagram (`Idol_SaaS_Full_System.puml`, 1851 lines) describing the intended system across three layers — Platform Admin, Tenant/Idol, and Fan — plus a set of formal design patterns (Command, Facade, Strategy, Decorator, Chain of Responsibility, Observer, State, Repository). Comparing it against the actual code, migrations, `idol_saas_api.yaml`, and `Checklists/`:

### 10a. Entire domains in the diagram with no implementation at all

These have **no model, no migration, no route, and no mention in any Checklist step** — they exist only as UML classes, and (for three of them) as unused OpenAPI schema components with no paths attached:

- **Media & Discography domain** — `MediaGallery`, `MediaItem`, `Discography`. Present as schemas in `idol_saas_api.yaml` (lines ~224–290) but never wired to a route, controller, or table. There is currently no way for a tenant to publish a photo/video gallery or a discography/release list.
- **External Sites domain** — `ExternalSite`, `ExternalSiteClick`. Also present as an unused schema in the YAML. This is distinct from the existing `SocialLink` model: the diagram's version is a generic, typed, click-tracked link list (streaming, ticketing, official site, fanclub, merch, YouTube, podcast, etc.) with analytics (`ExternalSiteClick`), whereas the current `SocialLink` model only covers social-platform profile links with follower counts — no click tracking, no arbitrary link types, no "linktree"-style page.
- **`ImpersonationSession`** — the diagram models impersonation as a first-class, auditable entity (`start()`, `end()`, `isExpired()`, tied to `adminId`/`targetTenantId`/`targetUserId`). The actual implementation (`Admin\TenantController@impersonate`) just mints a short-lived Sanctum token on the fly — there's no persisted session record, so there's no way to list active impersonations, force-end one early, or audit "who impersonated whom, when" beyond whatever ends up in `audit_logs` (if anything is logged there at all — not confirmed).
- **`SystemConfig`** — a generic platform-wide key/value settings store. No equivalent table/model exists; anything like this today would have to live in `.env` or be hardcoded.

### 10b. Modeled in the diagram but only partially implemented

- **Tenant-level RBAC** (`TenantMembership`, `Role` with `permissions: List<TenantPermission>`) — the diagram describes a real membership/role system where a tenant can have multiple users with different granular roles (`OWNER`, `ADMIN`, `BILLING`) and per-role permission lists. The actual implementation only has a single boolean (`users.is_tenant_admin`) — an admin either can or can't manage the tenant; there's no owner-vs-billing distinction, no custom roles, no permission granularity. Same gap on the platform side: `ServiceAdmin` has a `role` column and the diagram gives it `permissions: List<AdminPermission>`, but no permissions table/enum was found — `EnsureServiceAdmin` only checks "is this a ServiceAdmin and is it active," not fine-grained permissions like `MANAGE_PLANS` vs `VIEW_AUDIT_LOGS`.
- **`TenantConfig`** — diagram gives tenants a dedicated config entity with `brandingLogo`, `brandingColors`, `customDomain`, and an extensible settings map. The real `tenants` table has a single freeform `config` JSON column and no `customDomain` field anywhere on `Tenant` (only `PublishedSite.domain`, which is a different thing — the published output, not the tenant's chosen custom domain). This lines up with the "custom domains" and "tenant settings management" gaps already noted in `Refactor.md`.
- **`Plan.features: List<Feature>`** — the diagram gives each `Plan` a list of first-class `Feature` entities. In the real schema, feature gating is done entirely through the separate `FeatureFlag` + `feature_flag_tenant` pivot mechanism, not through anything attached to `Plan`. So plans and feature flags are two disconnected systems today, whereas the diagram intends them to be related (a plan grants a feature set).
- **Order/Subscription state machines** (`State Pattern — Order Lifecycle`, `State Pattern — Subscription Lifecycle`) — the diagram formalizes order and subscription status transitions as swappable `OrderState`/`SubscriptionState` classes. The real code just stores `status` as a plain enum/string column on `MerchOrder`/`FanSubscription` — functionally similar end state, but with no enforced transition rules, so nothing stops an invalid transition (e.g. `DELIVERED` → `PENDING`) at the model layer.
- **Payment methods** — `PaymentMethod` enum lists `CREDIT_CARD, PAYPAL, STRIPE, BANK_TRANSFER`; the actual webhook routes support a longer, different list (Stripe, PayPal, PromptPay, DuitNow, QRIS, PayNow, InstaPay, WeChatPay, Alipay) but only Stripe/PayPal have real SDKs installed — so the diagram and the real breadth of "supported" payment methods don't match in either direction.

### 10c. Architectural patterns in the diagram with no code equivalent

The diagram spends roughly half its length (~900 lines) on formal design-pattern scaffolding around the admin layer: **Command** (`AdminCommand`, `AdminCommandInvoker`, undo/redo history), **Strategy** (per-role admin operation strategies, payment strategies, content-visibility strategies), **Facade** (`AdminFacade`/`IAdminFacade`), **Decorator** (audit/permission/rate-limit decorators wrapping commands and facades), **Chain of Responsibility** (`AdminRequestHandler` chain: auth → permission → MFA → execute), **Observer** (`FanEventPublisher`/`FanEventListener` driving notifications, emails, and audit logging off domain events), **State** (order/subscription lifecycles), and **Repository** (`TenantAwareRepository` and per-entity repositories for blog, merch, fans, subscriptions, idol profiles).

None of this exists in the actual codebase. Controllers call Eloquent models and a handful of plain service classes (`CartService`, `CheckoutService`, `NotificationService`, `VisibilityGateService`, the `Payment/*WebhookHandler` classes) directly — there's no command bus, no repository layer, no facade, no request-handler chain, and no event/observer system wiring domain actions (new post, new merch, subscription expiring) to notifications or audit logs. This matters concretely: `FanEventPublisher` in the diagram is exactly the piece that would drive "notify fans when a new post/merch/event drops," which ties directly back into the queue/email gaps already flagged in §9 — right now there is no code path that turns a tenant admin's action into a fan notification or email automatically.

### 10d. Fully covered — no gap

The diagram's **Idol Profile**, **Blog**, **Fan & Membership**, **Merch/Commerce**, **Event & Schedule**, and **Fan Notification** domains all map cleanly onto existing models, migrations, and routes (`IdolProfile`, `IdolGroup`, `SocialLink`, `BlogPost/Category/Comment/Like`, `Fan`, `FanSubscription`, `MembershipTier`, `TierPerk`, `MerchProduct/Category/Variant/Cart/CartItem/Order/OrderItem`, `Address`, `Payment`, `Shipment`, `IdolEvent`, `EventRsvp`, `FanNotification`, `NotificationPreference`). `AuditLog` and `FeatureFlag`/tenant-override also exist and match. This is the bulk of the diagram and it's solidly built.

### 10e. Net takeaway

The diagram reads like the **target end-state architecture** (three fully realized domains plus a textbook enterprise pattern layer), while the actual code is a **pragmatic, working subset**: full commerce/content/membership functionality, but no media gallery, no discography, no generic external-links-with-analytics, no granular RBAC, no persisted impersonation trail, no plan↔feature linkage, and none of the Command/Strategy/Facade/Observer/Repository scaffolding — that layer was designed but never built. None of these gaps are mentioned anywhere in `Checklists/`, so if they're still wanted, they'd need new checklist steps (an "18", "19"... series) rather than being hidden work already in progress.
